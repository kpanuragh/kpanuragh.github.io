A lot of my friends have asked me, "Why do you love open source so much?" Honestly, for me, the answer is pretty simple: if something doesn’t work, I can fix it. That’s it. I mean, how great is that?

As a developer, I use tons of packages daily, many of which are built by other people. Of course, these packages aren’t always perfect—they might not follow the best coding practices, or they might have bugs. But here's the thing: as long as the package meets my needs and doesn’t have any show-stopping bugs, I can use it.

However, when something doesn’t work the way I need it to, open source gives me the freedom to take matters into my own hands. I can see the code and figure out how the developer tried to solve a problem. If it’s a small issue, I can fix it myself. I just fork the repo, make my changes, and submit a pull request (PR). Most of the time, they accept my changes, but even if they don’t, I can still use my version. And the best part? Others can benefit from my work too.

Let me share a few examples that really highlight why I love open source. One day, I contributed to a project called **NativePHP**, which helps developers create desktop applications with Laravel. When NativePHP was launched at Laracon India, I was super excited to try it out. However, when I tested it on Linux, I found a bug—Linux PHP binaries weren’t downloading properly for Electron, and it wasn’t detecting the operating system’s architecture. I managed to fix it, submitted a PR, and now my code is part of NativePHP. It felt amazing to contribute and see my work integrated into something I cared about.

Another example is when I was using **Waybar**, a status bar for my Sway desktop environment on Linux. I love how customizable it is, but I wanted it to display Spotify album art, and at the time, it didn’t support this feature. I wasn’t an expert in C++, but I understood enough to dive into the code, make some changes, and submit a PR. About a week later, the changes were approved, and now I can see my Spotify album art on my status bar. I remember thinking, "Wow, I actually did that!"

Even smaller contributions can be rewarding. I was using **Khal**, a terminal-based calendar utility, on Arch Linux and noticed that the installation instructions had issues. I forked the repo, fixed the documentation, and submitted a PR, which was quickly accepted. It wasn’t a massive change, but it still felt good knowing I helped improve the experience for others.

Now, I’m not a full-time open source contributor by any means, but I use open source software every day. Whenever I find a bug or an issue, I try to fix it. And if I can’t, I report it, and someone else usually steps in to help. The thing is, with proprietary software, you’re stuck. You can’t see what’s going on behind the scenes. You can’t fix issues yourself—you just have to wait for the company to release a fix. 

But with open source, you're empowered. You can look at the code, understand how it works, fix bugs, and make improvements. If there’s a bug and you need an immediate fix, you don’t have to wait for the next release—you can just clone the repo, build the software yourself, and you’re good to go. That’s the beauty of it.

One of the best feelings in the open source world is getting your first PR accepted. It’s a small but powerful reminder that you contributed to something bigger than yourself. When you fix an issue, or make a meaningful change, and know that people around the world are using software with *your* contribution—it’s hard to describe how fulfilling that is. It’s like this quiet moment of achievement that stays with you.
